### 2. Assumptions / interpretations

- **A1 â€” Command surface / naming (CONFIRM with user).** The existing `crafter statusline` subcommand RENDERS the panel (reads the Claude Code payload on stdin, prints the segment). A settings-editing mode must not overload it. This plan proposes a dedicated **`crafter install`** command group with two subcommands: **`crafter install statusline`** (runs the collision decision tree + `.bak`) and **`crafter install hook`** (registers the SessionStart hook). Rationale: groups the two installer-only mutations under one verb that reads as "installer machinery," keeps them clearly separate from the user-facing render command, and follows the existing grouped-subcommand pattern (`crafter buffer uat`/`gap`, `crafter skillbook add`/`get`/`init`). The JSON read/mutate/write + the "ours" matcher + `.bak` helper live in a shared internal package (proposed `cli/internal/claudesettings`, name is the Implementer's call) reused by both subcommands. **Alternative considered:** flat top-level commands `crafter install-statusline` / `crafter install-hook` â€” rejected as the primary because it pollutes the top-level namespace with hyphenated verbs unlike any existing command. **Flag this naming to the user before Phase 1** â€” it is the most visible design choice and cheap to change now, expensive later.
- **A2 â€” Interactive boundary / decision protocol (CONFIRM with user).** Bash owns TTY detection and the y/n prompt; the binary is strictly non-interactive. Proposed protocol: the statusline subcommand takes a flag that carries the *already-resolved* foreign-collision decision, e.g. `--on-foreign=overwrite|keep` (default `keep`), plus the inputs it needs (settings path, freshly-computed command). Bash computes interactivity, prompts via `/dev/tty` only when a real terminal is attached, and passes `--on-foreign=overwrite` only after a "yes". This makes the destructive path reachable in Go unit tests by simply passing `overwrite`, and in `tests/test_install.sh` by an internal env/flag the bash layer honors â€” with **no real TTY**. **Alternative considered:** a classify-then-apply two-step where the binary prints a machine-readable classification (absent/ours/foreign) and bash decides + re-invokes to apply. Rejected as the primary for being chattier and requiring two binary calls, but it is a clean fallback if the user prefers the binary to never decide. **Flag this protocol to the user before Phase 2.** Hard requirement either way: `curl | bash` (script body on stdin) must never be classified interactive â€” bash must consider `[ -t 1 ]`/`[ -t 2 ]` and/or `/dev/tty` openability, not just `[ -t 0 ]`, and read the prompt from `/dev/tty`.
- **A3 â€” "Ours" detection.** A `statusLine` is ours when its `command` is recognizably the crafter `statusline` invocation (the crafter binary path followed by the `statusline` subcommand). A composite wrapper that merely *embeds* the crafter command among other commands (the very compose-wrapper this feature used to emit as guidance) is FOREIGN, not ours. The exact matcher (anchored prefix/suffix vs. structural) is the Implementer's choice within this boundary; it must true-positive the plain crafter command and true-negative a `bash -c '... | <crafter> statusline ...'` wrapper.
- **A4 â€” "Differs" for rung 2.** The freshly-computed command always reflects the current install's binary path. Rung 2 updates when the stored command is ours but not byte-identical to the freshly-computed one (covers a moved binary path); identical â†’ no write at all.
- **A5 â€” `.bak` naming/hygiene.** The backup is a sibling of the settings file (e.g. `settings.json.bak`). Do not blindly clobber a pre-existing `.bak`; pick an idempotent, re-runnable scheme (Implementer's choice â€” e.g. skip writing a new `.bak` if one already preserves a foreign value, or suffix). Requirement: a foreign overwrite never destroys a prior recoverable backup, and re-running is safe.
- **A6 â€” Non-interactive foreign output is behavior-preserving.** The compose-wrapper / manual-merge guidance text and its escaping (the existing Node output produced a `bash -c '...'` composite with POSIX single-quote escaping of embedded commands) must be reproduced by the Go path closely enough that the existing escaping regression tests (G5, G6) still pass: the printed guidance is valid JSON whose `.statusLine.command` references both the existing command and the crafter `statusline` invocation, and an existing command containing single quotes yields a syntactically valid composite. The exact byte layout may differ as long as those assertions hold; the Implementer reproduces the two-layer (JSON + shell single-quote) escaping in Go.
- **A7 â€” Hook edit is behavior-preserving.** The Go hook subcommand reproduces the current Node logic: tolerant read (missing/invalid file â†’ start from empty object), ensure `hooks.SessionStart` is an array, append `{ hooks: [{ type: "command", command: <hookCmd> }] }` only if no existing entry already has that exact `command`, write `JSON.MarshalIndent(.., "", "  ")` + trailing `\n`. Note the hook always targets `$HOME/.claude/settings.json` even in `--local` mode (current behavior â€” preserve it; do not "fix" it in this task).
- **A8 â€” Opt-in default unchanged.** `--with-statusline` stays off by default; `--global`/`--local`, download/build, and symlink logic are untouched.
- **A9 â€” Test harness needs a real binary (load-bearing for Phase 2/3).** The existing `_make_fake_go_bin_dir` shim produces a binary that only `echo`s â€” it cannot mutate JSON. Today's hook/statusline install tests pass because *Node* did the mutation; after the port, the settings-edit assertions require a working `crafter` binary at `<dest>/crafter/bin/crafter`. The default `_run_installer` path (no fake shim) source-builds the real binary via the resolved Go toolchain (the test env has Go via mise + `.tool-versions` seeding), so those tests get a real binary. The migration must (a) keep the settings-edit tests on the real-binary path, and (b) for any test that injects the fake `go` shim, either not assert on settings mutation or pre-place a real binary at the destination. The Implementer picks the mechanism (e.g. a harness helper that builds/locates a real `crafter` once and seeds it at the destination, mirroring how `_make_fake_go_bin_dir` is injected). Requirement: install tests that assert on mutated settings run against a real mutation-capable binary, and the suite stays runnable without network access. **Flag if building a real binary in the suite proves too slow or unavailable in CI** â€” then a cached/pre-built binary path is needed.

### Alternatives considered

- **Keep mutation in `node -e` (the prior plan's node-mirroring approach).** Superseded by the user's accepted scope change â€” `node` is not in the stack; the binary already exists before both edits; Go is unit-testable and removes an undeclared installer dependency.
- **Flat top-level commands `crafter install-statusline` / `crafter install-hook`** (vs. the chosen `crafter install` group). Rejected as primary â€” hyphenated top-level verbs are unlike any existing command; a grouped verb matches the `buffer`/`skillbook` precedent. Cheap to switch; flagged for user confirmation (A1).
- **Classify-then-apply two-step protocol** (binary prints classification, bash decides + re-invokes) vs. the chosen single-call `--on-foreign` resolved-decision. Rejected as primary for being chattier and double-invoking, but kept as a clean fallback if the user wants the binary to never decide (A2).
- **Blind replace on `--with-statusline`.** Rejected by the agreed design â€” silently destroys third-party status lines, unacceptable under `curl | bash`.
- **Real-TTY simulation in install tests (pty / `/dev/tty` redirection).** Unnecessary now â€” the Go binary is non-interactive and the decision is injected, so the destructive path is reached deterministically without a TTY.
- **Reuse the echo-only fake go shim for settings-edit tests.** Rejected â€” the fake binary cannot mutate JSON; settings-edit assertions must run against a real binary (A9).

### 8. Risks / unknowns / flags

- **Command-surface naming (CONFIRM, A1).** `crafter install statusline` / `crafter install hook` vs. flat `crafter install-statusline` / `crafter install-hook`. Most visible decision; cheap now, costly post-release. Confirm with the user before Phase 1.
- **Interactive decision protocol (CONFIRM, A2).** Single-call `--on-foreign=overwrite|keep` vs. classify-then-apply two-step. Confirm before Phase 4 (and ideally before Phase 1, since the command flags are shaped in Step 1.3).
- **TTY detection under `curl | bash` (highest behavioral risk).** Script body is on stdin, so naive `[ -t 0 ]` is wrong; a bad predicate either hangs CI or never prompts a real user. Mitigation: Phase 4 detection considers `[ -t 1 ]`/`[ -t 2 ]`/`/dev/tty`, reads the prompt from `/dev/tty`, and Step 4.3 proves a piped invocation never prompts. Flag if the environment cannot reliably distinguish the cases.
- **Test harness needs a real binary (A9, central testability risk).** The echo-only fake go shim cannot mutate JSON; settings-edit tests must run against a real `crafter`. The default source-build path supplies one in the dev env, but flag if building a real binary in the suite is too slow or unavailable in CI â€” then a cached/pre-built-binary mechanism is needed. Also confirm the suite's no-hard-`node`-dependency intent survives de-noding the test-side parsing in G3/G5/G6.
- **Behavior-preserving hook port (A7).** The Go output must match Node's settings shape (indent, trailing newline, SessionStart structure) and idempotency, and must keep targeting HOME's settings even in local mode. Drift here breaks existing hook tests; covered by Phase 3 unit + install tests.
- **"Ours" matcher boundary (A3).** Must match the crafter command but not a composite wrapper embedding it, or an upgrade could silently replace a user's compose wrapper. Covered by Phase 1 unit tests; flag if the binary path can vary in ways that make a robust matcher ambiguous.
- **Escaping equivalence for guidance (A6).** The two-layer JSON+shell escaping is subtle; G5/G6 are the guard. The Go reproduction must keep the guidance valid JSON and single-quote-safe; flag if any G5/G6 assertion cannot be met without changing the test's intent.
- **`.bak` hygiene under repeated runs (A5).** A re-run must not clobber the original recoverable backup nor error; the exact scheme is the Implementer's, but must be idempotent â€” flag if no clean scheme fits the settings layout.

## Decisions

- **Decision (User Accepted):** Collision policy is a decision tree, not a blind replace â€” auto-install when absent; idempotent update when the existing line is ours; for a foreign line, prompt interactively (overwrite â†’ `.bak` + print old value) and fall back to non-destructive guidance when non-interactive. **Reason:** Preserves the "crafter panel is standalone, GSD not used" intent while never silently destroying a third-party status line, especially under `curl | bash`.
- **Decision (User Accepted):** When an overwrite of a foreign value occurs, persist the old value via a settings `.bak` AND print it. **Reason:** Recoverability for a destructive operation.
- **Decision (User Accepted):** Scope expanded â€” move ALL `settings.json` mutation out of `node -e` and into the Go `crafter` binary, for BOTH `install_statusline()` AND `install_hook()`; remove the `node` dependency from `install.sh` entirely. **Reason:** `node` is not in the project stack (PROJECT.md: Markdown + Go + Bash + cobra); the `crafter` binary already exists before both settings edits run; doing the JSON mutation + decision tree + `.bak` in Go aligns with the stack and makes the logic unit-testable in Go (which also resolves the original A1 interactive-test-reachability concern â€” the binary is non-interactive and the bash layer owns TTY detection/prompting).
- **Decision (User Accepted):** Command surface = nested group `crafter install statusline` / `crafter install hook` (planner assumption A1 confirmed). **Reason:** Matches the buffer/skillbook subcommand precedent and avoids overloading the existing `crafter statusline` render command.
- **Decision (User Accepted):** Interactive protocol = single binary call with a resolved-decision flag `--on-foreign=overwrite|keep` (planner assumption A2 confirmed); bash owns TTY detection + prompt, the binary stays non-interactive. **Reason:** Simplest; keeps the destructive foreign-overwrite path reachable in tests without a real TTY.
- **Decision (User Accepted):** Phase 1 review findings #2 + #3 deferred to **Phase 4**. **#2:** tolerant `Load` overwrites a malformed-but-non-empty settings file with no `.bak` on the Save paths â€” Phase 4 should weigh a backup-before-clobber on the foreign/overwrite path. **#3:** `Backup` skip-if-exists could let a stale unrelated `.bak` shadow a genuine foreign value Phase 4 is about to overwrite â€” Phase 4 overwrite body should weigh a content-aware/timestamped scheme. **Reason:** Both are most relevant where the destructive overwrite + `.bak` are actually exercised (Phase 4), not in the Phase 1 foundation. - **Decision (Orchestrator Accepted):** Step 1.1 â€” the `claudesettings` writer marshals object keys ALPHABETICALLY (Go `encoding/json` on a `map[string]json.RawMessage`), whereas the old Node `JSON.stringify` preserved insertion order.
