# Task: Progressive on-demand workflow loading for crafter-do

## Metadata

## Request
The real end goal recorded in `.crafter/STATE.md` `## Planned`, now unblocked because the Slices 1–7 decomposition is complete (every workflow step is a `rules/do/*` module; `skills/crafter-do/SKILL.md` is a thin manifest of pointer stubs + an upfront loader list).

**Goal:** Convert `skills/crafter-do/SKILL.md` from a loader that reads ALL `rules/do/*` modules upfront (the "Read and follow these rules" list currently enumerates the 4 core rules + all 15 `do/*` modules, so every invocation pays the full-workflow context cost) into a thin **orchestrator master-plan** that loads each step module **lazily — only when the orchestrator actually reaches that step**. The orchestrator holds only the step-sequence map (which step runs when, how they connect, the flag/scope branching) and Reads `rules/do/step-N.md` at the moment it enters that step, not before. Each step is an orchestrator-invoked unit, not user-invoked.

**This is NOT a move-and-link slice and NOT byte-identical:** it deliberately changes WHEN modules are loaded (lazy vs. upfront). Behavior of each step, once loaded, must stay identical; only the loading discipline changes.

Explicit design questions / decisions the plan MUST work out and surface (with competing interpretations where genuine):
1. How the orchestrator is instructed to lazily Read `rules/do/step-N.md` at step entry instead of via the upfront loader list — prompt-design only; no Go/CLI/mechanism change unless explicitly justified.
2. Which modules MUST stay eager (e.g. `flag-validation`, `project-resolution`, `step-0-resume` run before anything and gate entry; core rules `core.md`/`do-workflow.md`/`delegation.md`/`task-lifecycle.md`) vs. which can be lazy (`step-1`..`step-9b`).
3. The resume path — `step-0-resume` routes by plan status to a specific step; lazy loading must still let resume jump straight to the right module.
4. Cross-module references (stubs/modules reference sibling steps by name) must still resolve under lazy loading.
5. The `{CRAFTER_HOME}` runtime-path convention and the installer / `tests/test_install.sh` must still hold.
6. How to keep behavior-identical per-step while changing load timing, and how to verify it across the four flows (default, `--fast`, `--auto`, resume).
7. Whether the loader list is trimmed to only the eager core + a master-plan section replaces the stub list, OR stubs stay as the lazy-load trigger points.

Headline risks: master-plan correctness; resume-jump correctness.

Constraints: incremental, conservative, independently-verifiable plan mirroring the slice discipline that worked for the decomposition (pre-flight analysis gate, small phases, per-phase verifier + reviewer, behavior-identical-per-step where applicable). Out of scope: extension-skill composability (separate later concern), any Go/CLI change unless the plan explicitly justifies it as unavoidable.

**Scope update (2026-05-19, User Accepted — see `## Decisions`):** the architecture is **Option A agent delegation, CI-1 = Option C (hybrid: existing role-agents + a thin step-runner for glue steps)**. Path-normalization scope is **expanded**: this task now absorbs the **repo-wide `~/.claude/...` → `{CRAFTER_HOME}` normalization mandate** (formerly deferred to sibling task 20260421) plus the minimal install-time substitution. Sibling task 20260421 stays active and keeps the full build/transform pipeline + multi-runtime adapters; only its normalization sub-mandate transfers here. ### 1. Complete request

Redesign `skills/crafter-do/SKILL.md` from a manifest that **`Read`s all 4 core rules + 15 `do/*` modules upfront** into a thin **master-plan orchestrator** that holds only the step-sequence/navigation map (step order, scope branching, flag branching, resume entry points) and, **for each workflow step, delegates that step's procedure to an agent in an isolated subagent context that returns a structured summary** (Crafter's established Agent/Task pattern, `rules/delegation.md`). This delivers both end goals: (a) it removes the `Read(~/.claude/crafter/**)` per-file permission-prompt friction (Spike Q1), and (b) it gives true per-invocation context reduction because each step's procedure runs in an isolated subagent context, not the orchestrator's (Spike Q2). Option B (step-skills invoked by name) is rejected — skill-from-skill runs inline and would not reduce context cost.

**Delegation mechanism is RESOLVED as CI-1 = Option C (hybrid).** Steps that map to an existing role-agent are delegated to that agent, with the step's `rules/do/step-N.md` module as the agent's task brief; orchestration-glue steps with no existing role-agent are delegated to a single new thin step-runner agent that receives "step id + context bundle" and internally Reads/follows the right module. The exact bucketing is a Phase-0 deliverable, but the working partition (verified against the modules + `rules/delegation.md` + `agents/`) is:

- **Existing role-agent steps:** Step 2 → `crafter-analyzer` (discuss/research delegation already uses it); Step 3 → `crafter-planner`; Step 4 → `crafter-implementer`; Step 5 + Step 5a → `crafter-verifier`; Step 6 → `crafter-reviewer`.
- **Step-runner steps (new thin agent, no existing role):** `flag-validation`, `project-resolution`, `extension-skills`, Step 0 (resume), Step 1 (scope/completeness), Step 6a (session break), Steps 7–9 (post-change), Step 9b (PR composition). Phase 0 confirms or adjusts each placement and decides whether some of these stay fully orchestrator-side (little/no delegable body) rather than going to the step-runner.
- **Orchestrator-only bindings (ALWAYS inline in SKILL.md, never delegated regardless of which agent runs the delegable core):** Step 3 draft→approved `Edit` on the task file; Step 6 STOP + 5-iteration fix-loop cap and the (a)/(b)/(c) user choice; Step 6b approval-path selection + the commit; Steps 7–9 5-item mandatory checklist + the hard "do not end until all 5 addressed" gate; Step 9b no-standalone-`git push` invariant + escape-hatch handling; all user interaction; the plan-status/checkbox source-of-truth; the branch-sanity/main-master guards. This split is the headline architectural constraint.
